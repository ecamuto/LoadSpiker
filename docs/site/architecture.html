Architecture

A deep dive into LoadSpiker's layered architecture: from the Python API through the C extension to the protocol implementations.

Overview

Python API / CLI / Python Scripts
└── Python C Extension (python_extension.c)
    └── C Engine Core (engine.c)
        ├── HTTP via libcurl
        ├── WebSocket (websocket.c)
        ├── TCP (tcp.c)
        ├── UDP (udp.c)
        ├── MQTT (mqtt.c)
        └── Database (database.c)

LoadSpiker is structured in three distinct layers:

  • Python Layer: The user-facing API (loadspiker package), CLI entry point, scenario builders, assertions, session management, and authentication. All user code lives here.
  • Python C Extension: A thin bridge (python_extension.c) compiled as a shared library (_loadspiker.so). It translates Python calls into C function invocations, converts Python objects to C structs, and releases the GIL during long-running load phases.
  • C Engine Core: The performance-critical heart of LoadSpiker. Manages a connection pool, a worker thread pool (POSIX pthreads), and routes requests to the appropriate protocol implementation.

C Engine Core (engine.c)

Connection Pool

The engine maintains a bounded pool of CURL handles (CURLM multi-interface) managed by a mutex. Connections are leased to worker threads and returned after each request. The pool size is configured by max_connections.

Worker Thread Pool

Worker threads are created at engine initialization using POSIX pthread_create. Each thread pulls work items from a shared queue protected by a mutex and condition variable. The thread count is configured by worker_threads.

// Thread pool work item
typedef struct {
    engine_t       *engine;
    request_t      *request;
    response_t     *response;
    pthread_mutex_t done_mutex;
    pthread_cond_t  done_cond;
    int             completed;
} work_item_t;

Protocol Routing

Requests are dispatched to protocol handlers via engine_detect_protocol(), which inspects the URL scheme:

protocol_type_t engine_detect_protocol(const char *url) {
    if (strncmp(url, "http://",  7) == 0 ||
        strncmp(url, "https://", 8) == 0)  return PROTOCOL_HTTP;
    if (strncmp(url, "ws://",   5) == 0 ||
        strncmp(url, "wss://",  6) == 0)   return PROTOCOL_WEBSOCKET;
    if (strncmp(url, "tcp://",  6) == 0)   return PROTOCOL_TCP;
    if (strncmp(url, "udp://",  6) == 0)   return PROTOCOL_UDP;
    if (strncmp(url, "mqtt://", 7) == 0)   return PROTOCOL_MQTT;
    if (strncmp(url, "db://",   5) == 0)   return PROTOCOL_DATABASE;
    return PROTOCOL_HTTP; // default
}

Generic Request / Response Structures

All protocol handlers share a common request_t / response_t structure with a protocol-specific union for extra parameters:

typedef struct {
    char            url[MAX_URL_LENGTH];
    char            method[16];
    char           *headers[MAX_HEADERS];
    int             header_count;
    char           *body;
    size_t          body_length;
    int             timeout_ms;
    protocol_type_t protocol;
    union {
        mqtt_params_t     mqtt;
        tcp_params_t      tcp;
        udp_params_t      udp;
        database_params_t db;
    } protocol_params;
} request_t;

typedef struct {
    int    status_code;
    char  *body;
    size_t body_length;
    char  *headers[MAX_HEADERS];
    int    header_count;
    double response_time_ms;
    char   error[256];
} response_t;

Metrics Collection

Metrics are accumulated using atomic increments (__sync_fetch_and_add / _Atomic) for lock-free performance. Response times are accumulated via a running sum and count, then divided on read.

Python C Extension (python_extension.c)

The extension is built with distutils / setuptools and linked against the compiled engine.c object. It exposes a PyMethodDef table:

Python MethodC FunctionDescription
create_enginepy_create_engineInitialize the C engine, return a handle capsule
destroy_enginepy_destroy_engineFree all engine resources
execute_requestpy_execute_requestSingle HTTP request, returns a Python dict
run_load_testpy_run_load_testFull load test, releases GIL, returns metrics dict
get_metricspy_get_metricsReturn current metrics as a Python dict
reset_metricspy_reset_metricsZero all metric counters
websocket_connectpy_websocket_connectInitiate WebSocket connection
websocket_sendpy_websocket_sendSend WebSocket message
websocket_closepy_websocket_closeClose WebSocket connection
Missing Python Bindings: The C implementations for TCP (tcp.c), UDP (udp.c), MQTT (mqtt.c), and Database (database.c) exist and compile correctly, but they are not yet exposed in python_extension.c. Only WebSocket is exposed beyond HTTP. This means calling engine.tcp_connect(), engine.mqtt_connect(), etc. from Python will raise AttributeError until the bindings are added.

Use the helper to check at runtime:

def _has_protocol_method(engine, method_name):
    """Check if a protocol method is available in the C extension."""
    return hasattr(engine, method_name)

from loadspiker import Engine
engine = Engine()

if _has_protocol_method(engine, 'tcp_connect'):
    tcp_id = engine.tcp_connect("example.com", 9000)
else:
    print("TCP not available — skipping TCP test")

udp.c UDP Real POSIX socket-based implementation. Python bindings pending.
mqtt.c MQTT Real Implements CONNECT, PUBLISH, SUBSCRIBE, UNSUBACK packets. Python bindings pending.
database.c Database Simulated Simulates connection pool overhead. No real DB drivers. Python bindings pending.

Threading Model

Worker Thread Pool

At initialization, worker_threads POSIX threads are created and blocked on a condition variable, waiting for work items. The main thread (or any Python thread calling run_scenario) enqueues work items and waits for completion. This decouples request dispatch from execution and allows true parallel I/O.

Mutex-Protected State

The following shared state is protected by dedicated mutexes:

  • engine->metrics_mutex — Protects the metrics counters during updates.
  • engine->pool_mutex — Protects the connection pool during lease/return operations.
  • websocket_state_mutex — Protects the global WebSocket connection table.
  • mqtt_connections_mutex — Protects the MQTT connection array.
  • tcp_connections_mutex — Protects the TCP connection array.

GIL Release

The extension calls Py_BEGIN_ALLOW_THREADS before entering the C load test loop and Py_END_ALLOW_THREADS after it completes. This releases the Python GIL, allowing other Python threads to execute concurrently while the C engine runs the load test. Result conversion back to Python objects happens after GIL reacquisition.

Metrics Accumulation

Each worker thread accumulates response time and counts into thread-local structures, which are merged into the global metrics at the end of each batch. This avoids contention on the metrics mutex during the hot path.

Memory Management

Cleanup Functions

Each protocol module exports a *_cleanup_all() function called from engine_destroy():

  • tcp_cleanup_all(engine) — Closes all open TCP sockets and frees descriptors
  • udp_cleanup_all(engine) — Closes all UDP endpoints
  • mqtt_cleanup_all(engine) — Sends MQTT DISCONNECT and closes all broker sockets
  • websocket_cleanup_all(engine) — Clears all WebSocket connection state
  • database_cleanup_all(engine) — Releases all simulated connection pool entries

Buffer Safety

All string copies use strncpy with zero-initialization of the destination buffer before copying. Formatted writes use snprintf with explicit size limits. No strcpy or sprintf calls exist in the codebase.

// Safe string copy pattern used throughout
memset(dest, 0, sizeof(dest));
strncpy(dest, src, sizeof(dest) - 1);
// dest is always null-terminated

// Safe formatted write
char error_buf[256];
snprintf(error_buf, sizeof(error_buf),
         "Connection refused: %s:%d", host, port);

Credential Zeroing

MQTT passwords are zeroed from memory immediately after the CONNECT packet is sent, preventing credentials from persisting in process memory:

// After CONNECT packet is sent in mqtt.c
memset(params->password, 0, strlen(params->password));

malloc Checks

Every malloc / calloc call is followed by a NULL check. On allocation failure, the error is propagated up through the return value and surfaced to Python as a MemoryError.

Request Data Flow

  1. Python call: User calls engine.execute_request(url, method, headers, body).
  2. C Extension: py_execute_request parses Python arguments, builds a request_t struct, and enqueues a work item. GIL is released.
  3. Protocol Routing: engine_detect_protocol(request->url) returns the protocol type.
  4. Protocol Handler: The work item is dispatched to the appropriate handler (e.g., engine_execute_http_request for HTTP/HTTPS).
  5. Execution: The handler performs the actual I/O (libcurl for HTTP, raw sockets for TCP/UDP, etc.) and fills a response_t struct.
  6. Metrics Update: Response time and counters are atomically updated in the engine's metrics structure.
  7. GIL Reacquired: The C extension reacquires the GIL after the operation completes.
  8. Python Dict: py_execute_request converts response_t fields into a Python dict and returns it to the caller.
