Contributing

Thank you for your interest in contributing to LoadSpiker. This guide covers everything you need to set up a development environment, understand the codebase, follow code style guidelines, and submit a pull request.

Setup Steps

1

Fork and clone

git clone https://github.com/YOUR_USERNAME/loadspiker.git
cd loadspiker
git remote add upstream https://github.com/loadspiker/loadspiker.git
2

Install system dependencies

make install-deps
3

Complete developer setup (deps + build + install + test)

make dev-setup
4

Verify everything works

make build
make test
python3 -c "import loadspiker; print('OK')"

Key Directory Descriptions

  • src/ — All C code. Changes here require a make build to take effect.
  • loadspiker/ — Python package. Changes here are effective immediately when installed in editable mode (pip install -e .).
  • tests/ — Test suite. All tests use pytest. Protocol tests that require unavailable C bindings use @pytest.mark.skipif.
  • docs/site/ — Static HTML documentation. Edit the HTML files directly — no static site generator is required.

Development Workflow

Building and Testing

# Clean rebuild
make clean && make build

# Install in editable mode
make install
# or:
pip install -e .

# Run the full test suite
make test
# or:
python3 -m pytest tests/ -v

# Run a specific test file
python3 -m pytest tests/test_engine.py -v

# Run tests matching a keyword
python3 -m pytest tests/ -k "session" -v

Debug Builds

# Build with debug symbols (-O0 -g)
make debug

# AddressSanitizer build (Linux)
CFLAGS="-fsanitize=address -g" LDFLAGS="-fsanitize=address" make build

# Run tests with AddressSanitizer
make test-asan

# Valgrind (Linux only)
valgrind --leak-check=full --error-exitcode=1 \
  python3 -m pytest tests/test_engine.py -v

Code Style Guidelines

C Code Style

  • 4-space indentation, no tabs
  • Doxygen-style comments on all public functions
  • Always check malloc/calloc return values for NULL
  • Use const for pointer parameters that are not modified
  • Use strncpy and snprintf — never strcpy or sprintf
  • Zero-initialize buffers before copying into them
  • Opening brace on the same line as the function signature (K&R style)
/**
 * @brief Connect to an MQTT broker.
 *
 * @param engine     Pointer to the engine instance.
 * @param host       Broker hostname or IP address.
 * @param port       Broker port number (typically 1883).
 * @param client_id  Unique client identifier string.
 * @return           Connection ID >= 0 on success, -1 on error.
 */
int mqtt_connect(engine_t *engine,
                 const char *host,
                 int port,
                 const char *client_id) {
    if (!engine || !host || !client_id) {
        return -1;
    }

    char safe_host[MAX_HOSTNAME_LENGTH];
    memset(safe_host, 0, sizeof(safe_host));
    strncpy(safe_host, host, sizeof(safe_host) - 1);

    mqtt_connection_t *conn = calloc(1, sizeof(mqtt_connection_t));
    if (!conn) {
        return -1;  // Allocation failure
    }

    // ... implementation
    return conn->id;
}

Python Code Style

  • Follow PEP 8 — use flake8 or ruff to check
  • Add type hints to all public function signatures
  • Write docstrings for all public classes and methods (Google or NumPy style)
  • Use f-strings for string formatting
  • Maximum line length: 100 characters
  • Sort imports: stdlib, then third-party, then local (isort)
from typing import Optional, Dict, List


class SessionStore:
    """Thread-safe key-value store for a single virtual user's session state.

    Args:
        user_id: Unique identifier for this user's session.
    """

    def __init__(self, user_id: str) -> None:
        self.user_id = user_id
        self._data: Dict[str, object] = {}
        self._cookies: Dict[str, str] = {}

    def set(self, key: str, value: object) -> None:
        """Store a value under the given key.

        Args:
            key:   The storage key.
            value: The value to store (any serializable type).
        """
        self._data[key] = value

    def get(self, key: str, default: Optional[object] = None) -> Optional[object]:
        """Retrieve a stored value.

        Args:
            key:     The storage key.
            default: Value to return if the key does not exist.

        Returns:
            The stored value, or default if not found.
        """
        return self._data.get(key, default)

Unit Tests

Unit tests should be fast (under 100ms each), require no network access, and mock external dependencies. Place them in tests/test_*.py files. Use pytest fixtures for shared setup.

import pytest
from loadspiker.assertions import status_is, body_contains, run_assertions


@pytest.fixture
def sample_response():
    return {
        'status_code': 200,
        'body': '{"id": 1, "name": "Alice"}',
        'headers': {'Content-Type': 'application/json'},
        'response_time_ms': 42.0,
        'error': None,
    }


def test_status_is_pass(sample_response):
    assertion = status_is(200)
    passed, message = assertion(sample_response)
    assert passed


def test_status_is_fail(sample_response):
    assertion = status_is(404)
    passed, message = assertion(sample_response)
    assert not passed
    assert "200" in message

Integration Tests

Integration tests interact with the C engine and may make real network requests. Mark tests that require network access with @pytest.mark.integration. Use pytest -m "not integration" to run only unit tests in CI.

Performance Tests

Performance tests live in test_benchmarks.py and should assert minimum throughput thresholds. These are slow and should only run in dedicated benchmark jobs, not in regular CI.

# Run all tests except benchmarks
python3 -m pytest tests/ -v --ignore=tests/test_benchmarks.py

# Run only benchmarks
python3 -m pytest tests/test_benchmarks.py -v

# Run with ASan
make test-asan

# Run all test types
make test-all

Contributing Process

  1. Check issues: Search the GitHub Issues page to see if your intended change is already being discussed. Comment on an existing issue or open a new one before starting significant work.
  2. Fork the repository: Click "Fork" on GitHub and clone your fork locally.
  3. Create a branch: Branch off main with a descriptive name. 
    git checkout -b feature/tcp-python-bindings
# or
git checkout -b fix/mqtt-memory-leak
  4. Make your changes: Write code following the style guidelines above. Keep commits focused — one logical change per commit.
  5. Add or update tests: All new functionality must have corresponding tests. Bug fixes should include a regression test.
  6. Open a pull request: Push your branch and open a PR against main. Fill in the PR template, describing what changed and why.

PR Checklist

Before submitting your pull request, confirm all of the following:

  • Code follows the C and Python style guidelines described above
  • All existing tests pass: make test
  • New or changed functionality has test coverage
  • Memory is checked with ASan or Valgrind for C changes: make test-asan
  • Documentation is updated if the public API changed
  • No compiler warnings: build with -Wall -Wextra and fix all warnings
  • Thread safety is maintained — all shared state is protected by appropriate mutexes
  • Error handling is present — all C functions return errors, Python raises exceptions
  • Input validation is performed before use — null pointers, empty strings, out-of-range values
  • No hardcoded secrets, credentials, or environment-specific paths in the code

Memory Safety (Critical for C Code)

LoadSpiker's C engine is security-sensitive. Pay particular attention to the following classes of vulnerabilities in any C code you write or modify:

  • Buffer overflows: Always use strncpy(dest, src, sizeof(dest) - 1). Always zero-initialize destination buffers with memset before writing. Never use strcpy, strcat, sprintf, or gets.
  • Null pointer dereferences: Validate all pointer arguments at the top of every function. Return an error code immediately if a required pointer is NULL.
  • Memory leaks: Every malloc/calloc must have a corresponding free on every code path — including error paths. Use the *_cleanup_all() pattern for connection tables.
  • Use-after-free: Set pointers to NULL immediately after calling free(). Never access memory through a pointer after the object it points to has been destroyed.
  • Thread safety: Acquire the appropriate mutex before reading or writing any shared state. Use atomic operations for simple counters where a mutex would be too expensive. Document which mutex protects which data structure.
Tip: Run make test-asan before submitting any C code change. AddressSanitizer catches buffer overflows, use-after-free, and double-free errors at runtime. A clean ASan run is required for C code PRs.
