Roadmap & Changelog

LoadSpiker's development phases, planned features, and version history.

Phase 1: Foundation COMPLETED — Q1 2025

  • C engine core with libcurl-based HTTP/HTTPS support
  • Python C extension (python_extension.c) with full HTTP API surface
  • Worker thread pool (POSIX pthreads) with configurable concurrency
  • Connection pool with per-handle CURL management
  • Multi-protocol architecture: WebSocket, TCP, UDP, MQTT, Database stubs
  • WebSocket support (Phase 1 simulation — connection overhead measurement)
  • MQTT C implementation: CONNECT, PUBLISH, SUBSCRIBE, DISCONNECT packets
  • TCP C implementation: socket-based connect, send, receive, disconnect
  • UDP C implementation: endpoint create, send, receive, close
  • Database C implementation: simulated connection pool with configurable latency
  • Python Scenario API: get(), post(), put(), delete(), add_request()
  • Data-driven testing with CSV support: sequential, random, round-robin strategies
  • Assertion system: status_is, body_contains, json_path, body_matches, header_exists, custom_assertion, AssertionGroup
  • Performance assertions: throughput_at_least, avg_response_time_under, error_rate_below, success_rate_at_least, max_response_time_under
  • Session management: thread-safe SessionStore, cookie handling, token storage
  • Request correlation: JSON path, header, cookie, and regex extraction rules
  • Authentication flows: Basic, Bearer, API Key, Form-based, OAuth 2.0, Custom
  • Reporter system: ConsoleReporter, JSONReporter, HTMLReporter, MultiReporter
  • Load utilities: ramp_up(), spike_test(), stress_test(), constant_load()
  • CLI interface with load pattern syntax, interactive mode, and config file support
  • Built-in benchmarks achieving 10,000+ req/s on commodity hardware
  • Comprehensive test suite: unit, integration, and performance tests
  • Memory-safe C code: strncpy/snprintf everywhere, malloc NULL checks, cleanup functions

Phase 2: Advanced Testing Capabilities PLANNED

2.1 Distributed Testing

  • Master-worker architecture for coordinating load across multiple machines
  • Docker Compose configuration for spinning up distributed test clusters
  • Kubernetes operator for cloud-native load test orchestration
  • Synchronized scenario execution and centralized metrics aggregation
  • Automatic worker discovery via DNS or static host lists

2.2 Advanced Load Patterns

  • Extended data sources: JSON, database queries, API feeds as scenario data
  • Conditional request logic based on previous response values
  • Variable think times sampled from statistical distributions (normal, exponential)
  • Per-user weight factors for heterogeneous scenario mixes

2.3 Dynamic Scaling

  • Auto-scaling worker count based on measured CPU utilization and queue depth
  • Feedback-controlled load generation to hold a target RPS
  • Graceful scale-down with in-flight request completion

Phase 3: Enterprise Reporting PLANNED

3.1 Advanced Analytics

  • Real-time dashboards with WebSocket-streamed live metrics
  • P50, P75, P90, P95, P99, P99.9 response time percentile calculations
  • Apdex score computation and SLA compliance reporting
  • Response time distribution histograms and heatmaps
  • Comparative analysis: side-by-side comparison of multiple test runs
  • Trend analysis over time across repeated CI/CD runs

3.2 CI/CD Integration

  • Jenkins plugin with JUnit XML output and build failure on SLA breach
  • GitLab CI template and native performance test artifact upload
  • GitHub Actions marketplace action with PR comment posting
  • Prometheus metrics endpoint for real-time scraping
  • Grafana dashboard JSON templates for instant visualization setup
  • Slack and PagerDuty alerting integration

Phase 4: User Experience PLANNED

4.1 Web-Based GUI

  • React or Vue web application for scenario authoring without writing code
  • Visual load shape designer (drag-and-drop ramp, spike, step patterns)
  • Real-time test monitoring dashboard accessible from any browser
  • Test library: save, version, and reuse scenario configurations

4.2 Developer Tooling

  • VS Code extension with scenario syntax highlighting and inline metrics
  • JetBrains plugin for PyCharm and IntelliJ
  • CLI enhancements: shell completion for Bash, Zsh, Fish
  • HAR file import: generate scenarios from browser recordings
  • Browser extension for recording and exporting test scenarios
  • OpenAPI / Swagger spec import for automatic scenario scaffolding

Phase 5: Enterprise Features PLANNED

5.1 Security Testing

  • OWASP Top 10 load testing profiles (injection, auth bypass, etc.)
  • TLS/SSL certificate validation and mTLS client certificate support
  • Fuzzing mode: automatically mutate request payloads for robustness testing
  • Authentication bypass probing with safe, non-destructive test patterns

5.2 Massive Scale

  • 1,000,000+ concurrent virtual user support via distributed agents
  • Geo-distributed load injection from multiple cloud regions
  • Bandwidth simulation and network condition modeling (latency, packet loss)
  • Long-duration soak testing (24h+) with memory leak detection integration

Changelog

[Unreleased]

Added

  • Python bindings for TCP, UDP, MQTT, and Database protocol methods (in progress)
  • WebSocket real-message exchange via libwebsockets integration (planned)
  • P95 / P99 percentile tracking in the C engine metrics layer
  • make test-asan target for AddressSanitizer test runs

Fixed

  • Memory leak in MQTT CONNECT packet builder when password was NULL
  • Race condition in WebSocket connection table on concurrent close operations
  • UDP receive returning stale data when buffer was not zeroed between calls
  • HTTP response header parsing failing on headers with colons in values

Changed

  • Improved engine_detect_protocol() to return PROTOCOL_HTTP as default instead of error
  • Session cookie storage now uses a sorted list for deterministic header ordering
  • JSONReporter now outputs ISO 8601 timestamps

Security

  • MQTT passwords are now zeroed from memory immediately after CONNECT packet transmission
  • All string copies in C code now use strncpy with explicit size bounds

[1.0.0] — 2025 Q1

Initial Release

  • C engine core with libcurl HTTP/HTTPS support and connection pooling
  • Python C extension with full API surface
  • Multi-protocol architecture (HTTP real, WebSocket/Database simulated)
  • Scenario API: Scenario, RESTAPIScenario, WebsiteScenario, MQTTScenario
  • Assertion system: response assertions and performance assertions
  • Session management with thread-safe SessionStore and request correlation
  • Authentication: Basic, Bearer, API Key, Form-based, OAuth 2.0, Custom
  • Reporter system: Console, JSON, HTML, Multi
  • CLI with interactive mode, load patterns, config files
  • Data-driven CSV testing with sequential/random/round-robin strategies
  • Complete test suite with unit, integration, and performance tests
  • Build system: Makefile with install-deps, build, test, debug, benchmark targets

Known Issues

Missing C Extension Protocol Bindings: TCP, UDP, MQTT, and Database C implementations exist and compile, but are not yet exposed as Python methods. Tests for these protocols are skipped automatically. Track progress in Architecture: Python C Extension.
