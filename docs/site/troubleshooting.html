Troubleshooting

Solutions to common problems encountered when installing, building, or running LoadSpiker.

Segmentation Fault on Startup

Symptoms

  • Python crashes immediately with Segmentation fault (core dumped)
  • Crash occurs when importing loadspiker or creating an Engine() instance
  • Stack trace shows a crash inside _loadspiker.so

Likely Causes

  • Mismatched Python versions between build time and runtime
  • Stale .so file from an older build — rebuild required
  • Incompatible libcurl version (API mismatch)
  • Memory corruption in the C engine (use ASan to diagnose)

Solutions

Build with debug symbols and run under GDB:

make debug
gdb python3
(gdb) run -c "import loadspiker; e = loadspiker.Engine()"
(gdb) bt  # print backtrace

Build with AddressSanitizer (Linux):

CFLAGS="-fsanitize=address -g" LDFLAGS="-fsanitize=address" make build
python3 -c "import loadspiker; loadspiker.Engine()"

Build with AddressSanitizer (macOS):

CFLAGS="-fsanitize=address -g" \
LDFLAGS="-fsanitize=address" \
ASAN_OPTIONS=detect_leaks=1 make build
python3 -c "import loadspiker; loadspiker.Engine()"

Use Valgrind (Linux only):

valgrind --leak-check=full --error-exitcode=1 \
  python3 -c "
from loadspiker import Engine
e = Engine(max_connections=5, worker_threads=1)
e.destroy()
"

Clean rebuild to eliminate stale artifacts:

make clean
make build
make install

ImportError: No module named 'loadspiker'

Solutions

Add the project root to PYTHONPATH:

export PYTHONPATH=/path/to/loadspiker:$PYTHONPATH
python3 -c "import loadspiker"

If using a virtual environment, ensure it is activated:

source venv/bin/activate
python3 -c "import loadspiker"

Verify the compiled shared library exists:

ls -la loadspiker/_loadspiker*.so
# If missing, rebuild:
make clean && make build

Install in editable mode (recommended for development):

pip install -e .
python3 -c "import loadspiker; print(loadspiker.__version__)"

Build Failures

Missing libcurl

error: curl/curl.h: No such file or directory
# Ubuntu / Debian
sudo apt-get install -y libcurl4-openssl-dev

# CentOS / RHEL
sudo yum install -y libcurl-devel

# macOS
brew install curl
# Then tell pkg-config where to find it:
export PKG_CONFIG_PATH="$(brew --prefix curl)/lib/pkgconfig"

Missing Python Headers

error: Python.h: No such file or directory
# Ubuntu / Debian
sudo apt-get install -y python3-dev

# CentOS / RHEL
sudo yum install -y python3-devel

# Verify the correct version
python3-config --includes

Missing pkg-config

make: pkg-config: No such file or directory
# Ubuntu / Debian
sudo apt-get install -y pkg-config

# macOS
brew install pkg-config

High Memory Usage

If LoadSpiker consumes excessive memory during large-scale tests, try these mitigations:

  • Reduce max_connections: Each connection handle has a fixed buffer overhead. Halving max_connections roughly halves connection-pool memory. 
    engine = Engine(max_connections=50, worker_threads=4)
  • Reduce worker_threads: Each thread has a stack (typically 8 MB on Linux). Use fewer threads if memory is constrained. 
    engine = Engine(max_connections=100, worker_threads=2)
  • Avoid storing large response bodies: If you do not need the response body for assertions, do not log or accumulate it between requests.
  • Throttle request rate: Add a think_time in scenarios to reduce concurrent in-flight requests and the associated buffer allocations.

Low Performance / Low RPS

OS-Level Limits

# Check current file descriptor and process limits
ulimit -n    # open files (should be >= max_connections * 2)
ulimit -u    # processes

# Increase for the current session
ulimit -n 65536

# Permanent increase (add to /etc/security/limits.conf):
# * soft nofile 65536
# * hard nofile 65536
# Linux: check kernel TCP parameters
cat /proc/sys/net/ipv4/ip_local_port_range
# Should be a wide range, e.g. 1024 60999
# Expand if needed:
sudo sysctl -w net.ipv4.ip_local_port_range="1024 60999"
sudo sysctl -w net.ipv4.tcp_tw_reuse=1

Python-Level Profiling

# Profile your scenario script
python3 -m cProfile -s cumulative my_test.py | head -30

System Monitoring During Tests

# CPU and memory
top -p $(pgrep -f loadspiker)

# I/O wait
iostat -x 1

# Active connections
netstat -an | grep ESTABLISHED | wc -l
ss -s

Thread Safety Issues

Symptoms

  • Intermittent crashes or corrupted results when running tests concurrently
  • Sporadic double free or use-after-free errors detected by ASan
  • Metrics show wildly inconsistent values between runs

Solutions

  • Use one Engine instance per test run: The engine is designed to be used from a single controlling thread. Do not call run_scenario from multiple Python threads simultaneously on the same engine. 
    # WRONG: concurrent access from multiple threads
import threading
engine = Engine()
threads = [threading.Thread(target=engine.run_scenario, args=(s,)) for s in scenarios]

# CORRECT: separate engine per thread
def run_in_thread(scenario):
    eng = Engine(max_connections=50)
    eng.run_scenario(scenario, users=10, duration=30)
    eng.destroy()

threads = [threading.Thread(target=run_in_thread, args=(s,)) for s in scenarios]
  • Do not share SessionStore objects: Each virtual user must have its own session store. See Multi-User Session Isolation.

Protocol-Specific Errors

AttributeError for TCP / UDP / MQTT / Database

AttributeError: '_loadspiker.LoadSpikerEngine' object has no attribute 'tcp_connect'

This error means the C extension does not yet expose Python bindings for the requested protocol. The C implementation files (tcp.c, udp.c, mqtt.c, database.c) exist and compile correctly, but the PyMethodDef entries have not yet been added to python_extension.c.

Test suite skip markers: Tests for these protocols are decorated with @pytest.mark.skipif so the test suite passes even when the bindings are absent. This is intentional and expected behavior, not a test failure. See Architecture: Python C Extension for the full list of exposed methods.

Check availability at runtime:

from loadspiker import Engine

engine = Engine()
protocols_available = {
    'tcp':      hasattr(engine, 'tcp_connect'),
    'udp':      hasattr(engine, 'udp_create_endpoint'),
    'mqtt':     hasattr(engine, 'mqtt_connect'),
    'database': hasattr(engine, 'database_connect'),
}
print(protocols_available)
# {'tcp': False, 'udp': False, 'mqtt': False, 'database': False}
# HTTP and WebSocket are always available

Debug Mode

Python Logging

import logging
logging.basicConfig(
    level=logging.DEBUG,
    format='%(asctime)s %(levelname)s %(name)s: %(message)s'
)

from loadspiker import Engine, Scenario

engine = Engine(verbose=True)  # Enable C-level verbose output
scenario = Scenario("Debug Test")
scenario.get("https://httpbin.org/get")
results = engine.run_scenario(scenario, users=2, duration=10)

C Engine Debug Build

# Build with LOADSPIKER_DEBUG=1 for extra C-level logging
LOADSPIKER_DEBUG=1 make build

# Or set at runtime (if debug logging is compiled in)
LOADSPIKER_DEBUG=1 python3 my_test.py

# Verbose C output
LOADSPIKER_VERBOSE=1 python3 my_test.py

Memory Testing with ASan

# Run tests with AddressSanitizer
make test-asan

# Run a specific test with ASan
CFLAGS="-fsanitize=address -g" make build
python3 -m pytest tests/test_engine.py -v
